# `niro/harness/` — the test harness

This directory holds the **harness**: a standard, scriptable lifecycle —
`start` / `stop` / `seed` / `reset` — for the application under test.

Niro authors and runs these scripts for THIS project (a human can run them directly
too). Everything else works against the **running target** they produce — the fixed
verbs mean the app stands up, seeds, and resets **the same way regardless of the
app**, so any consumer, agent or human, gets a clean, reproducible baseline.

The scripts are the reproducibility guarantee — the whole harness must rebuild from
them, from scratch.

## The lifecycle interface

Expose these four verbs. Two rules on *how* you write them:

- **`seed` is written in the application's own language** — reuse the app's
  factories / seed command / migrations, or drive its API with the app's
  standard-library HTTP client. The app's runtime is already present and the team
  owns that language, so don't pull in a foreign one (no `seed.py` for a Go app).
- **`start` / `stop` / `reset` are thin orchestration** — build, serve, tear down.
  Write them for the **host you're actually on** (or as container commands); keep
  them simple and portable so adding another OS later is a small step, but **don't
  author an untested script for an OS you're not running** — a `.ps1` written on
  Linux and never executed proves nothing. Add the other OS when you're on it.
  (Containerized — the preferred path — it's Linux inside regardless.)

| Verb | Must do |
|---|---|
| `start` | Build the **current working-tree** code and serve it; bring up the full service graph (app + DB + cache + workers); healthcheck every surface. |
| `stop`  | Tear the running target down cleanly. |
| `seed`  | Populate a **deterministic** baseline — the same logical actors/data every time, so a reset reproduces the same scenario. Emits/refreshes `../credentials.yaml` and `../fixtures.yaml`. |
| `reset` | Restore the clean baseline: wipe runtime state and re-seed (restore a golden snapshot where possible, else re-run `seed`). The clean-baseline primitive called between runs. |

## A time budget for the verbs

Design the verbs so wall-clock tracks the **real work** they do. As a reference,
`stop` / `seed` / `reset` should finish in **under a minute**; `start` may take
longer only for a genuine build, and `reset` should prefer restoring a golden
snapshot over a full re-seed when it can.

Treat a verb that runs **far longer than its actual work** as a defect to
diagnose, not a cost to accept — the usual cause is the command *waiting* on
something (a background process that never detached and holds the caller's output
open, a poll loop, a hung pipe), not honest computation. Never meet the budget by
cutting correctness or hiding the delay — no blanket `timeout` wrappers, no
skipped healthchecks or verification. Find why the clock exceeds the work, and fix
that.

Verify it once, when you author the verbs: run each under a timeout (e.g.
`timeout 90 ./reset.sh > log 2>&1`) and confirm it returns well under budget — a
verb that doesn't is the defect above (a command that detaches a process but
leaves it holding the caller's output will hang here, not when redirected to a
file). Run `reset` twice and confirm it lands the same clean baseline both times.
This is a one-time authoring check, not something every run repeats.

## Where state lives

- **`niro/harness/` (committed)** — the scripts above plus any harness config.
  Small, version-controlled; they *are* the rebuild-from-scratch guarantee.
- **`niro/harness/run/` (gitignored)** — **all** mutable runtime state: built
  binaries, the database + WAL, logs, the golden snapshot. Regenerable; never
  committed.

## Prefer a container — keep the host clean

The harness should leave **no footprint on your machine**. The cleanest way to
guarantee that is to run the app **inside a container built from the working
tree**: the whole runtime — process, database, logs, build output — lives in the
container, so `stop`/`reset` tears it down and leaves your host and source
untouched. In effect, the container *is* `run/`.

- **Use the app's own `Dockerfile` / `docker-compose` if it has one** — the most
  faithful way to run it, and the least work. A multi-service app stays
  multi-container via its compose (app + DB + cache), not crammed into one.
- **Build from the working tree — never a published or prebuilt image.** Otherwise
  the harness serves stale code, which defeats the whole exercise.
- **Running natively on the host is a fallback** — for when containerizing is
  genuinely impractical (no Dockerfile and an awkward build). Then the rule below
  carries the host-cleanliness guarantee instead.

## The read-only-source rule

Container or host, the rule is the same: treat the project source (everything
outside `niro/`) as **read-only for runtime state**. No database, logs, or build
artifacts you control are written into the source tree — all of it goes under
`harness/run/` (or stays inside the container). A build tool may write its own
caches/deps in-tree; that's its business — the harness itself leaves no runtime
footprint in the code under test. This keeps the app under test pristine and makes
`reset` a matter of wiping `run/` (or the container), never cleaning the source.
